# Project Post-Mortems (v3 Development)

Per the post-mortem lifecycle rule (`.claude/rules/07-documentation.md`), an entry here is
_incident (≤3 lines) → operationalization shipped (rule/hook/guard link)_. The point of a
post-mortem is not the narrative but what got operationalized in its wake; narrative beyond
that is deleted — full incident write-ups live in git history (pre-2026-07 versions of this
file). An entry with no operationalization outcome is a red flag to fix, not a keepsake.

> **Scope**: this file holds ALL v3 post-mortems. `CLAUDE.md`'s Post-Mortems table lists only
> the catastrophic, AI-specific behavioral failures (unrecoverable loss of code, data, or
> context) and links here for full details. Universal principles from these incidents were
> promoted to `~/.claude/CLAUDE.md`.

## Design Decision: Why v3 Abandoned DDD

DDD was over-engineered for a one-person project: circular dependency issues, excessive
abstraction layers, complex bootstrap/wiring — more time spent fixing architecture than
building features. v3 uses simple classes, constructor injection, and clear service
boundaries (`.claude/rules/01-architecture.md`). Ship features, not architecture.
