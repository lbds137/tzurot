- 🏗️ `[LIFT]` **Validate `/wallet/list` response against Zod schema at api-gateway boundary** — Surfaced 2026-04-26 by claude-bot review on PR #908. The canonical `ListWalletKeysResponse` schema declares `timestamp: z.string()` (required) and per-key `createdAt` / `lastUsedAt`. The api-gateway handler at `services/api-gateway/src/routes/wallet/listKeys.ts:51-59` hand-constructs the response shape and currently emits all required fields — so no observed runtime bug. But the type contract is enforced by convention, not runtime check: a future refactor that drops `timestamp` (or any other required field) would silently break bot-client consumers using `ListWalletKeysResponse` without typecheck or test catching it. **Fix shape**: call `ListWalletKeysResponseSchema.parse(payload)` before `sendCustomSuccess` (or wrap `sendCustomSuccess` so route handlers pass the schema and parse happens centrally). Same gap applies to other gateway endpoints with hand-coded responses against schemas in `common-types/schemas/api/`. Folds naturally into the Observability & Telemetry theme's API-contract concerns. Surfaced 2026-04-26.

- ✨ `[FEAT]` **User-facing notice when partial attachment failures occur** — Beta.106's hotfix (PR #893) lets the conversation proceed when only some attachments succeed (deliberately better than aborting the whole thread on one bad URL). Side effect: a user forwarding a 5-image post where 2 succeed and 3 fail will see the bot respond to only the 2 surviving images with no indication that 3 were dropped. The response may appear unexpectedly sparse. **Fix shape**: emit a lightweight ephemeral notice ("⚠️ 3 of 5 attachments couldn't be loaded — link expired or external host blocked") alongside the normal response, tied to the per-attachment failure list already produced by `DownloadAttachmentsStep`. Surface via `errorInfo`-style structured field on the success result (e.g. `partialFailures?: string[]`), then bot-client renders the notice. **Why deferred**: hotfix scope; need design call on ephemeral vs. inline placement and on per-failure detail vs. summary count. Surfaced 2026-04-25 by claude-bot review on PR #893 (minor).
