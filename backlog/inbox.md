- 🐛 `[FIX]` **TTS chunker produces oversized chunk that voice-engine rejects (off-by-six)** — Surfaced 2026-04-27 by Lilith observation; characterized via beta.107 prod logs. Failure trace from deployment `433bbeb5-2e3d-4d80-bec2-96dad2ddb29a` at 02:08 UTC: ElevenLabs TTS timed out (60s, `ElevenLabsTimeoutError`) → fallback to voice-engine → cold-start polling for 44s → multi-chunk synthesis with `textLength=2983, chunkCount=2` → **second chunk was 2006 characters; voice-engine returned 400 "Text too long (2006 chars). Maximum: 2000"** → `withRetry` correctly classified as non-retryable (4xx) → text-only response delivered (no TTS audio).

  **Root location**: `services/ai-worker/src/services/voice/ttsSynthesizer.ts` `splitTextIntoChunks()`. Static read of all branches in `accumulateSentence` and `forceSplitLongSentence` _appears_ to constrain output to ≤ `MAX_CHUNK_LENGTH (2000)`, but production produced 2006. There's a subtle path (regex edge case? trim() interaction? template literal join arithmetic?) the chunker hits that I haven't pinpointed without the original Lilith response text.

  **Fix shape**: at minimum, add a defensive truncation at the chunker output: `chunks.map(c => c.length > MAX_CHUNK_LENGTH ? c.slice(0, MAX_CHUNK_LENGTH) : c)` before returning. Cuts the failure path even if root-cause analysis stalls. Then root-cause it properly via property-based testing or a reproducer derived from the failing text.

  **Adjacent issues**:

  **(a) ElevenLabs single-attempt retry (beta.97 cut 2→1)** — both TTS failures observed in beta.107 logs hit the 60s ElevenLabs timeout on attempt 1 with no retry. With the old `MAX_ATTEMPTS=2`, a transient ElevenLabs hiccup would retry once instead of falling all the way through to the cold-start voice-engine path (~44s). Reverting to `=2` increases worst-case TTS latency by 60s on persistent failure but improves user-visible success rate on transient failure. Worth measuring telemetry first: how often does attempt 1 fail and the voice-engine fallback succeed (= retry would have helped) vs. attempt 1 fail and voice-engine also fails (= retry wouldn't change outcome).

  **(b) Two chunker implementations with same 2000-char cap** — `splitMessage()` in `common-types/utils/discord.ts:198` splits for Discord's message limit (preserves code blocks, natural-boundary splitting); `splitTextIntoChunks()` in `ai-worker/services/voice/ttsSynthesizer.ts:70` splits for voice-engine's TTS limit (sentence-boundary regex, force-split fallback). Both cap at 2000 chars; both split at natural boundaries; different algorithms. **Harmonization opportunity**: extract a shared chunking primitive in common-types parameterized by (boundary-strategy, code-block-preservation, max-length). Each caller wraps with their own pre/post-processing.
