- 🧹 `[CHORE]` **`wrapUrlsForNoEmbed` cuts URLs at internal `]` (RFC 3986 path-segment chars)** — Surfaced 2026-05-01 in PR #954 round 1 (claude-review). The regex+callback approach added by PR #954 walks the URL candidate cutting at the first `]` or `>` outside a balanced paren group. The original regex `[^\s<>)]+?` _admitted_ `[` and `]` inside URLs (RFC 3986 allows them in some contexts, e.g., IPv6 literals like `https://[::1]/path` or array-index query params); the new walker terminates on `]`. So `https://api.example.com/items[0]/value` would render as `<https://api.example.com/items[0>]/value` — same shape of bug as the original `)` issue this PR fixed. **Why deferred**: `]` inside a URL path is very uncommon in practice and doesn't appear in any provider error message we've observed. Reviewer self-classified as "minor" + explicit "shouldn't block merge — just worth a backlog note if you want full RFC 3986 compliance later." **Fix shape**: extend the paren-balancing walker to also balance `[`/`]` pairs (mirror the depth-tracking logic). ~5 LOC change + 2-3 tests. **Promote when**: a real URL containing `]` surfaces in production error messages, OR opportunistically when next touching `wrapUrlsForNoEmbed`. Surfaced 2026-05-01 in PR #954.
