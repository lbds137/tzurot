- 🏗️ `[LIFT]` **`tryResolveUserKey` no-key negative caching + invariant test** — Surfaced 2026-04-27 in PR #924 rounds 7+8 (claude-review). For `zai-coding`-with-no-key (the auto-fallthrough majority path), `tryResolveUserKey` returns `null` without cache write → every request re-reads DB. Currently accepted overhead documented in code comment at `services/ai-worker/src/services/ApiKeyResolver.ts`; resolveApiKey's `source: 'system'` semantics can't be reused because `zai-coding` has no system fallback. **Fix shape (caching)**: introduce a distinct cache state for "no user key configured" (e.g., a `source: 'absent'` discriminant or sentinel value), populate it from `tryResolveUserKey`'s null branch, and short-circuit subsequent reads through the same entry-point. Touches `ApiKeyResolver`'s cache shape; likely needs co-update to `resolveApiKey` callers that peek the cache directly. **Adjacent test gap (round 8)**: even before changing the caching, the documented "accepted overhead — do NOT cache" invariant has no test. Add a `tryResolveUserKey` unit test that asserts cache miss + DB null → null result, AND that a second call re-reads the DB (no sentinel cached). Reviewer-provided test body is in PR #924 round-8 review. ~10 LOC test addition. **Promote when**: production logs show measurable per-request DB-read load on the fallthrough path, OR if a second auto-fallthrough provider lands and the duplicate-DB-read pattern compounds. The invariant test alone is a 5-min standalone task that locks in current behavior even if the broader caching change is deferred.
