- 🏗️ `[LIFT]` **PR 2 absorbs: z.ai integration follow-ups from PR #921 review rounds** — Surfaced 2026-04-27 across 6 review rounds on PR #921.   **Medium — explicit omit of `frequencyPenalty`/`presencePenalty` in `buildZaiCodingModel`** (rounds 5+6): `services/ai-worker/src/services/ModelFactory.ts` `buildZaiCodingModel` passes `shared.frequencyPenalty` and `shared.presencePenalty` to the `ChatOpenAI` constructor even though `filterRestrictedParams` always strips them to `undefined` for z.ai-direct routes (provider-tier filter).   **Medium — validation model hardcoding (`ZAI_VALIDATION_MODEL`)**: `packages/common-types/src/constants/ai.ts` hardcodes `'glm-4.5-flash'` as the probe model for z.ai key validation. If z.ai deprecates or renames it, every valid key will validate as `INVALID_KEY` (silent false-negative). **Fix shape options**: (a) try a list of fallback models in order (`glm-4.5-flash` → `glm-4.7-flash` → first-from-`/models`-endpoint); (b) hit a non-chat endpoint (e.g., `/account/balance` if z.ai exposes one — needs research); (c) accept the maintenance risk and add a periodic check to detect catalog changes early. **Why deferred to PR 2**: tied to the broader catalog handling needed for the `ProviderRouter` model-name translation logic.

  **Minor — `__includeRawResponse` absence comment in `buildZaiCodingModel`**: `buildOpenRouterModel` has a detailed comment explaining why `__includeRawResponse: true` is set (LangChain ↔ OpenRouter `message.reasoning` field-name bridging). `buildZaiCodingModel` omits the field with no explanation.   **Minor — `VALIDATION_MESSAGES.QUOTA_EXCEEDED_ZAI` consolidation**: `services/api-gateway/src/utils/apiKeyValidation.ts` introduces `VALIDATION_MESSAGES` to deduplicate generic error strings across validators, but `validateZaiCodingKey`'s `'z.ai coding-plan quota exhausted for the current period'` is still inline. **Fix shape**: add a `QUOTA_EXCEEDED_ZAI` entry to `VALIDATION_MESSAGES` for symmetry.

  **Medium — ai-worker validators classify 5xx as `InvalidApiKeyError`** (round 3): `services/ai-worker/src/services/KeyValidationService.ts`'s `validateElevenLabsKey` AND `validateZaiCodingKey` both throw `InvalidApiKeyError` for any non-2xx response, including 5xx. During upstream provider downtime, a correct key would be flagged as invalid in the runtime health check. **Fix shape**: add a separate arm for `response.status >= 500` that throws a transient/provider error class (likely a new class — `ProviderError` or similar). Touches both validators in one pass for consistency. Affects existing ElevenLabs path too — not new with z.ai. **Promote when**: production logs show false-positive InvalidApiKeyError during a provider outage, OR fold into PR 2 since it's adjacent to ProviderRouter's error-classification design.

  **Verification — z.ai's HTTP status for 402 (expired/lapsed subscription)** (round 3): 401/403/429 are accounted for. Folds into PR 2's broader UX considerations for the routing layer.
