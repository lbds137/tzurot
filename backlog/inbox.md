- 🐛 `[FIX]` **MultiTagRecovery background-run race with `startResultsListener` after timeout** — Surfaced 2026-05-16 during PR #1034 review. When recovery exceeds the 30s `Promise.race` timeout in `index.ts`, the background `recovery.run()` continues iterating through remaining entries. Each `adoptRehydratedEntry` call registers new jobIds in `jobToGroup`, but during the brief window between `chatManager.submitChatJob` returning a fresh jobId and `adoptRehydratedEntry`'s `jobToGroup.set` running, a result for that new jobId could arrive (via the already-started `startResultsListener`), find `ownsJob` false, and fall through to the single-personality path — bypassing slot ordering. **Low probability**: timeout implies degraded Discord, which also implies slow result delivery, narrowing the window. **Fix shape options**: (a) thread a cancellation token through recovery so the loop bails on timeout; (b) pre-populate `jobToGroup` per slot inside `rebuildSlot` immediately after `submitChatJob` (analogous to the `startFanOut` race fix); (c) accept the race and add a stronger comment. **Promote when**: if production logs show "recovery exceeded 30s" non-zero frequency, OR if the race surfaces as a missed-ordering bug.

Other buffered messages for the same channel might be delayed or stalled. **Fix shape**: write a focused integration test that exercises the all-terminal adoption path and asserts on `ResponseOrderingService`'s queue state post-flush. If a leak exists, either (a) skip `registerJob` when allDone is computable before registration, or (b) add an explicit cleanup call. **Promote when**: opportunistic during the next `ResponseOrderingService` touch, OR if recovery-related ordering issues surface in production logs.

- 🏗️ `[LIFT]` **`buildSentinelPersonality` type-safety hardening** — Surfaced 2026-05-16 during PR #1034 review. `MultiTagRecovery.buildSentinelPersonality` returns a `Partial<LoadedPersonality>` shape cast via `as unknown as LoadedPersonality` — only `id`/`slug`/`displayName`/`name` are populated. The deliverError path consumes those four fields; any future caller touching other fields (e.g., `llmConfig`, `systemPrompt`) will silently observe `undefined` rather than receive a type error. **Fix shape**: replace the cast with a discriminated-union sentinel type (e.g., `LoadedPersonality | { __sentinel: true; id, slug, displayName, name }`) so consumers must explicitly handle the sentinel case, OR change `LoadedPersonality` itself to make the four "always-present" fields non-optional and the rest optional + add a `isSentinel` discriminator. **Promote when**: a downstream caller of recovery-rebuilt slots needs to access a non-sentinel field of `LoadedPersonality` (next refactor of `SlotDeliveryService` or any extension to `deliverError`).

- 🐛 `[FIX]` **Recovery-time synthetic `personaId` causes silent FK violations in conversation history** — Surfaced 2026-05-16 during PR #1034 review. `MultiTagRecovery.rebuildSlot` builds slots with synthetic `personaId` strings (`recovery-persona-${id}`, `recovery-revoked-${slug}`, `recovery-denied-${id}`) for cases where the original `personaId` isn't recoverable from the snapshot. When `SlotDeliveryService.deliverError` runs `saveAssistantMessage(personaId, ...)`, the synthetic string fails the Prisma UUID FK constraint to `personas.id`. The error is caught by the try/catch (webhook already sent), so the user sees the error message — but conversation history doesn't record it. **Fix shape**: in `MultiTagRecovery.recoverOne`, resolve the user's default `personaId` via `PersonaResolver` (already available in the bot-client DI graph) and use that for sentinel slots instead of the synthetic strings. ~20 LOC + test. **Promote when**: recovery-slot count in production logs becomes non-negligible (currently zero — recovery only fires after a crash with in-flight multi-tag fan-outs, which is rare).
