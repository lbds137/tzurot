- 🧹 `[CHORE]` **Buffer-hash → in-memory LRU cache for TTS text recovery** — Council-suggested upgrade path during PR #955 design review. Today (post-PR-#955) the receive-side handler emits a placeholder text (`🔁 *Forwarded voice message originally spoken by \`{slug}\`...\*`) instead of the actual TTS source text. The placeholder is sufficient for the LLM context but loses fidelity vs. seeing the real utterance. **Fix shape**: at TTS synthesis time (`services/ai-worker/src/services/voice/`), compute a hash of the audio buffer (e.g. `sha256(buf).slice(0,16)`) and store `hash → originalText`in an in-memory LRU cache (single-instance bot-client deployment, so memory is fine;`lru-cache`is already a transitive dep). On the receive side in`VoiceTranscriptionService.handleOwnBotAudio`, after classifying the attachment as bot-authored, download the buffer (already done by the existing STT path), hash it, look up in the cache. Cache hit → use real text. Cache miss → fall back to the current placeholder. **Why deferred from PR #955**: the v1 placeholder is sufficient; the recovery cache requires plumbing across ai-worker → bot-client (where to populate? bot-client uploads, ai-worker synthesizes) and a download step we currently skip. **Promote when**: a user reports the placeholder feels too information-poor in practice, OR alongside any voice-engine refactor that naturally exposes the synthesis-side hash injection point. Surfaced 2026-05-01 in PR #955.

- 🧹 `[CHORE]` **`wrapUrlsForNoEmbed` cuts URLs at internal `]` (RFC 3986 path-segment chars)** — Surfaced 2026-05-01 in PR #954 round 1 (claude-review). The regex+callback approach added by PR #954 walks the URL candidate cutting at the first `]` or `>` outside a balanced paren group. The original regex `[^\s<>)]+?` _admitted_ `[` and `]` inside URLs (RFC 3986 allows them in some contexts, e.g., IPv6 literals like `https://[::1]/path` or array-index query params); the new walker terminates on `]`. So `https://api.example.com/items[0]/value` would render as `<https://api.example.com/items[0>]/value` — same shape of bug as the original `)` issue this PR fixed. **Why deferred**: `]` inside a URL path is very uncommon in practice and doesn't appear in any provider error message we've observed. Reviewer self-classified as "minor" + explicit "shouldn't block merge — just worth a backlog note if you want full RFC 3986 compliance later." **Fix shape**: extend the paren-balancing walker to also balance `[`/`]` pairs (mirror the depth-tracking logic). ~5 LOC change + 2-3 tests. **Promote when**: a real URL containing `]` surfaces in production error messages, OR opportunistically when next touching `wrapUrlsForNoEmbed`. Surfaced 2026-05-01 in PR #954.
