- 🧹 `[CHORE]` **Normalize non-standard IP formats in `publicRouteKeyGenerator`** — Surfaced 2026-05-17 by PR #1046 final review. Two edge cases where the same client lands in different Redis buckets depending on format: (a) **Bracket+port form** — `X-Forwarded-For: [2001:db8::1]:8080` (RFC 7239 §6.3 bracket+port; rare but valid in some Nginx configs). Current `startsWith('[') && endsWith(']')` check leaves brackets intact because `endsWith(']')` is false, producing key `ratelimit:public:[2001:db8::1]:8080` instead of `ratelimit:public:2001:db8::1`. (b) **IPv4-mapped IPv6** from `socket.remoteAddress` — Node exposes IPv4 clients via `::ffff:127.0.0.1` on some platforms. The same host's `127.0.0.1` direct connection would land in a different bucket. **Promote when**: Railway logs show non-standard XFF formats, or when adopting a non-Railway hosting environment. **Won't surface in current Railway deployment.** **Fix shape**: regex-based strip in the XFF path (`last.replace(/^\[([^\]]+)\].*$/, '$1')` to handle bracket+port), and an `::ffff:` prefix strip on `socket.remoteAddress`. Start: `services/api-gateway/src/utils/RedisRateLimiter.ts` `publicRouteKeyGenerator`.
