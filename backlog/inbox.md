- üßπ `[CHORE]` **Reconsider `ErrorCode.UNAUTHORIZED` ‚Üí HTTP 403 mapping (probably should be 401) and split from `forbidden()`** ‚Äî Surfaced 2026-05-18 by PR #1051 (when `vi.importActual` exposed the mock-vs-prod divergence). `services/api-gateway/src/utils/errorResponses.ts:20` maps `ErrorCode.UNAUTHORIZED` ‚Üí HTTP 403 (FORBIDDEN). Per RFC 7235, HTTP 401 = "lacks valid authentication credentials" ‚Äî fits missing/wrong `X-Service-Auth` better than 403 (which implies "authenticated but lacking permission"). **There's a related code smell**: `ErrorResponses.unauthorized` and `ErrorResponses.forbidden` BOTH wrap `ErrorCode.UNAUTHORIZED`, so a naive flip to 401 would also break `forbidden()` (currently returns 403 by virtue of the conflation). The right fix is a **paired change**: introduce a separate `ErrorCode.FORBIDDEN` variant (‚Üí 403), point `forbidden()` at that, then flip `ErrorCode.UNAUTHORIZED` ‚Üí 401 and `unauthorized()` keeps using it. **No current user-visible bug** ‚Äî clients don't behave differently between 401 and 403 in our context ‚Äî but RFC compliance + helper-vs-code parity are the right defaults. **Fix shape**: add `ErrorCode.FORBIDDEN` to the enum, map it to 403, switch `ErrorResponses.forbidden` to use it; change `[ErrorCode.UNAUTHORIZED]: 403` ‚Üí `[ErrorCode.UNAUTHORIZED]: 401`; audit tests that assert 403 on the auth-failure path. ~15-20 LOC + test updates. **Promote when**: any API consumer reports confusion between auth-required and auth-insufficient errors, or when next touching errorResponses.ts.
