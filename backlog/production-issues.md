## 🚨 Production Issues

_Active bugs observed in production. Fix before new features._

- 🐛 `[FIX]` **DM system prompt: non-active participants emitted with full `about_user` block** — Surfaced 2026-04-28 by user (live observation in prod). **Shipped in beta.110 (PR #932); awaiting in-prod verification.** Both dev and prod are now on beta.110. Verification step: trigger a fresh DM with the bot and confirm a new diagnostic dump shows no foreign `<about>` blocks in `<participants>`. Remove this entry once prod deploy ships and verification passes. Root cause: `loadPersonasAndResolveReferences` in `services/ai-worker/src/services/ConversationalRAGService.ts` was injecting personas resolved from personality static fields (e.g. `@[Name](user:UUID)` in `conversational_examples`) into the participants list. PR #932 removes the injection while preserving text-replacement. Trace confirmed this was the sole mechanism (chat_log `extractParticipants`, current-message `mentionedPersonas`, and `getAllParticipantPersonas` all clean for the dump's repro case). In a DM session between user A and a personality, the personality's owner (user B) is being assembled into the `<participants>` block of the system prompt as a non-active participant (`active="true"` is correctly set only on user A), but the inactive participant entry still includes the full `<about source="user_input">` body. The `about_user` field is not a public persona description — it is the persona owner's self-authored profile, intended for ai-worker's context when the OWNER chats with the personality. Surfacing it in another user's prompt causes (a) the LLM to conflate the active speaker's identity with the inactive participant (observed in `postProcessing.thinkingContent`: model reasons about the speaker as if they were the inactive participant, then synthesizes blended vocabulary across the two), and (b) any identity-bearing content in the about block reaches a user who shouldn't see it. **Trace start**: search `services/ai-worker/src/services/` for participant-assembly logic — the system prompt builder or conversation-context assembler. The `active` flag is computed correctly today (only the speaker is active), so the fix likely lives in either (i) suppressing inactive participants entirely in DM context, (ii) dropping the `<about>` block on inactive participants, or (iii) gating persona-owner inclusion on guild-membership context. **Severity**: high — the inactive participant's about block is verbatim user-authored content that the requesting user has no business seeing. **Repro signal in dump**: `meta.guildId === null` AND `assembledPrompt` contains a `<participant>` whose `id` differs from the `from_id` of every chat-log message AND whose `<about>` is non-empty. **Adjacency**: same DM-path-missing-a-scoping-filter shape as the link-resolution bug in `backlog/inbox.md`; possibly one fix lands both. Surfaced 2026-04-28.
