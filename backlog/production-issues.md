## 🚨 Production Issues

_Active bugs observed in production. Fix before new features._

- 🐛 `[FIX]` **Vision negative-cache poisons attachments forever on transient AUTH failures** — Surfaced 2026-04-28 by user (Lila/Vladlena) — observed prod log of `qwen/qwen3.5-397b-a17b` returning `401 Missing Authentication header`, while same key + same model succeeded for a different user 30 min earlier. **Fix in flight via this PR.** Root cause: error classification at `packages/common-types/src/constants/error.ts:208-214` unconditionally tags AUTHENTICATION as PERMANENT, which conflates retry policy (correct: fail-fast on bad keys) with cache policy (incorrect: don't remember an auth failure forever). Cascade: `withRetry` skips → `VisionDescriptionCache.storeFailure({permanent: true})` writes to L1 Redis (1h TTL) AND L2 PostgreSQL (NO TTL, no eviction path) → future reads hit L2 → repopulate L1 indefinitely → user gets `[Image unavailable: API key issue]` forever, with no manual recovery path. **Fix scope** (council-validated, Gemini 3.1 Pro Preview): (a) decouple cache policy from retry policy via per-category `VISION_FAILURE_CACHE_POLICY` map keyed on `ApiErrorCategory`; AUTH/QUOTA get 5min L1 cooldown, attachment-bound failures (CONTENT_POLICY/MEDIA_NOT_FOUND/MODEL_NOT_FOUND) get 60min, retryable-transient categories keep 10min; (b) drop L2 PostgreSQL cache entirely (over-engineering for ephemeral Discord attachments — Redis restarts are rare on Railway and the L2 protection created the poison-forever surface); (c) add diagnostic logging at the failure site (userId, apiKeySource, jobId, personalityName, provider, attachmentId, cachedAt); (d) source-aware fallback strings — system-key AUTH failure no longer says "API key issue" (which would blame the user); user-key AUTH failure points at `/wallet`. **Repro signal**: `[VisionDescriptionCache] Stored failure in negative cache permanent=true category=authentication` in ai-worker logs, followed by repeated cache HITs returning the same fallback for the same `attachmentId`. Remove this entry once prod deploy ships and a fresh AUTH-error attachment is observed to recover within 5 minutes (instead of staying poisoned).

- 🐛 `[FIX]` **DM system prompt: non-active participants emitted with full `about_user` block** — Surfaced 2026-04-28 by user (live observation in prod). **Fix merged via PR #932 (commit `c85bcdaa5`); awaiting prod deploy + verification.** Dev Railway auto-deploys develop, so the fix is on dev now; prod deploy is a separate manual decision. Verification step: trigger a fresh DM with the bot and confirm a new diagnostic dump shows no foreign `<about>` blocks in `<participants>`. Remove this entry once prod deploy ships and verification passes. Root cause: `loadPersonasAndResolveReferences` in `services/ai-worker/src/services/ConversationalRAGService.ts` was injecting personas resolved from personality static fields (e.g. `@[Name](user:UUID)` in `conversational_examples`) into the participants list. PR #932 removes the injection while preserving text-replacement. Trace confirmed this was the sole mechanism (chat_log `extractParticipants`, current-message `mentionedPersonas`, and `getAllParticipantPersonas` all clean for the dump's repro case). In a DM session between user A and a personality, the personality's owner (user B) is being assembled into the `<participants>` block of the system prompt as a non-active participant (`active="true"` is correctly set only on user A), but the inactive participant entry still includes the full `<about source="user_input">` body. The `about_user` field is not a public persona description — it is the persona owner's self-authored profile, intended for ai-worker's context when the OWNER chats with the personality. Surfacing it in another user's prompt causes (a) the LLM to conflate the active speaker's identity with the inactive participant (observed in `postProcessing.thinkingContent`: model reasons about the speaker as if they were the inactive participant, then synthesizes blended vocabulary across the two), and (b) any identity-bearing content in the about block reaches a user who shouldn't see it. **Trace start**: search `services/ai-worker/src/services/` for participant-assembly logic — the system prompt builder or conversation-context assembler. The `active` flag is computed correctly today (only the speaker is active), so the fix likely lives in either (i) suppressing inactive participants entirely in DM context, (ii) dropping the `<about>` block on inactive participants, or (iii) gating persona-owner inclusion on guild-membership context. **Severity**: high — the inactive participant's about block is verbatim user-authored content that the requesting user has no business seeing. **Repro signal in dump**: `meta.guildId === null` AND `assembledPrompt` contains a `<participant>` whose `id` differs from the `from_id` of every chat-log message AND whose `<about>` is non-empty. **Adjacency**: same DM-path-missing-a-scoping-filter shape as the link-resolution bug in `backlog/inbox.md`; possibly one fix lands both. Surfaced 2026-04-28.
