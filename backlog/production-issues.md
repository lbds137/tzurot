## 🚨 Production Issues

_Active bugs observed in production. Fix before new features._

- 🐛 **[FIX — ROOT CAUSE FOUND, fix pending] Forwarded-message-as-trigger conveys EMPTY content to the LLM** — Forwarding a message into an activated/@mention channel produces a generic reply because the bot submits **empty** message content. **ROOT CAUSE (confirmed via the #1171 diagnostic, 2026-06-08)**: NOT snapshots — the forward's snapshot content IS present at the gateway event (diagnostic showed `referenceType:1, snapshotSize:1, snapshotContentLen:338, detectedAsForward:true`). `getEffectiveContent` correctly extracts the 338 chars. The loss is in **`ReferenceExtractor.extractReferencesAndMentions` (`services/bot-client/src/services/contextBuilder/ReferenceExtractor.ts:119-124`)**: `updatedContent` (from `extractReferencesWithReplacement` → `ReferenceFormatter.format`) is computed by running Discord-link replacement on **`updatedMessage.content`** (the forward's empty top-level content), then unconditionally **clobbers** the authoritative `content`: `if (updatedContent !== undefined) messageContent = updatedContent` → empty wins. Works for normal messages only because there `message.content === content`; forwards diverge. **FIX**: thread the effective `content` into `extractReferencesWithReplacement` → `formatter.format` so link-replacement operates on the forwarded text, not `message.content` (no-op for normal messages; preserves forward content). Plus structural hardening: make the effective-content param REQUIRED so the silent `message.content` default can't recur. **NO SELF-HEAL / NO BACKFILL POSSIBLE**: the forward content exists ONLY on the live gateway `MESSAGE_CREATE`; a REST re-fetch (history/sync) returns empty snapshots, so already-persisted empty forwards are unrecoverable (confirmed: user observed no self-heal; sync shrink-skip log `originalLength=172 newLength=0` shows refetch returning empty). The trigger moment is the only capture point. **#1170 (`hydrateForwardedSnapshots`) addressed a non-existent problem** (snapshots aren't missing) and its `fetch(true)` premise is empirically false (REST lacks the snapshot) — REMOVE it. **#1171 diagnostic** stays until the real fix is dev-verified, then remove (tracked in quick-wins). **Shadow blind spot**: #1169 made the worker's `rawMessageContent` ground-truth-empty for forwards, so the legacy payload and worker agreed on empty → false `allMatched:true`; the shadow structurally can't catch this class. Originally surfaced 2026-06-07 (dev shadow burn-in).

_Cleared 2026-05-19:_

- _**Multi-personality ping race: wrong personality + duplicate response content** → resolved by **PR #1049** (shipped in v3.0.0-beta.123). Root cause: `ResponseOrderingService.processQueue` called a shared `deliverFn` for all drained results; the multi-tag closure ignored its arguments, so when two groups raced in the same channel one group's deliverFn re-delivered its own slots while the other group's deliverFn never fired. Fix: store `deliverFn` per `BufferedResult`. Originally observed 2026-05-17 in `#monotheism` channel — three @-pings to two personalities returned three Samael responses with the second text-identical to the first. Re-add this entry if the symptom resurfaces._

_Cleared 2026-05-10:_

- _**Cross-channel history bypassed `maxAge` and `contextEpoch` filters; cross-channel budget was a residual** → resolved by **PR #1011** (merged 2026-05-10, ships in next release). Three interlocking bugs: `getChannelHistory` ignored maxAge at the DB layer (stale rows leaked through, filling dbHistory to dbLimit); `fetchCrossChannelIfEnabled` computed cross-channel budget as `dbLimit - currentHistoryLength` (silently zero when current is full); `getCrossChannelHistory` had no time filter at all (would surface arbitrarily old context even after the other two fixed). Fix: shared `computeHistoryCutoff` helper used by both Prisma queries; cross-channel gets its own dbLimit budget (additive, not residual) — matches the user's mental model that cross-channel context is part of memory continuity, not "filler when current is sparse"._

- _**Voice transcription pipeline hang via `await channel.sendTyping()`** → resolved across **PR #1000 + PR #1001** (shipped in v3.0.0-beta.119). PR #1000 wrapped sendTyping in a fire-and-forget helper with latency telemetry + ESLint guard against re-introducing `await` on the call (root cause: discord.js REST queue stalls under sustained Discord rate-limit pressure, leaving the promise neither resolved nor rejected).
