- _**Small-context model overflows: full-window fill + no output headroom + tokenizer mismatch** → resolved by **PR #1186** (+ UI-text follow-up **#1187**), clamp dev-verified 2026-06-12 18:30 UTC (`configured=32768 modelContextLength=32768 effective=24576` on the Dolphin 32k model). Three compounding causes, all addressed: (1) validation exempted ≤64k models from the headroom cap (full context allowed) → shared `computeContextCap` in common-types now caps small models at 75% / large at 50%, enforced at gateway save time AND as a worker runtime clamp (`contextWindowResolver`: `min(configured, cap(real context_length))`, warn-log on engage, graceful pass-through when the model is unknown); (2) the budget filled the whole window with no output reserve → the cap's headroom now covers output + tokenizer mismatch; (3) tiktoken undercounts vs Mistral-family tokenizers ~15-25% → absorbed by the same margin. `ContentBudgetManager.allocate` takes the effective window as a required option (the `|| 131072` silent fallback is gone). Pre-existing oversized rows need no migration (runtime clamp protects); owners should still re-save affected presets to ≤cap to silence the per-generation warn. Follow-up tracked in inbox: the `content as string` token-count cast._

- 🐛 **[INVESTIGATE — symptom confirmed, mechanism unknown] Preset (llm-config) PUTs time out — gateway >10s on the update path** — Preset edit (section modal), global-toggle, and set-default commands fail with `"Request timeout (gateway slow or unavailable)"`; the user retries and it keeps failing. **Confirmed from prod logs (2026-06-12 ~01:36–01:54 UTC, admin + user paths)**: bot-client calls `PUT /api/admin/llm-config/{id}` and `PUT /api/user/llm-config/{id}` (+ `.../set-free-default`) with a **10s request timeout**; the gateway takes **>10s** so bot-client aborts (`updateGlobalPreset`/`updatePreset` in `services/bot-client/src/commands/preset/api.js`). Gateway logs the matching `request aborted … responseTime=10005/9999/9990… statusCode:null` — pinned at the 10s abort, request never completed. So it's **genuine gateway slowness on the llm-config update path**, not merely a too-short timeout (a config write should be sub-second). **Mechanism NOT yet known**: during those 10s the gateway logged NOTHING — prod `prisma:query` logging is off, no processing/error lines, no pool/lock/deadlock signals; a surface scan of the PUT handler showed no obvious heavy op. Intermittent (some config ops in the same window succeeded), which *suggests* resource contention (DB connection-pool wait / lock) rather than a constant slow query — **hypothesis, unconfirmed**. **Investigate — NOTE: not dev-reproducible.** It's load-correlated (prod has concurrent users across servers/DMs; dev is a single user with no load), so it must be observed in PROD, not reproduced in dev. Path: ship **targeted timing instrumentation to the prod llm-config PUT** (breakdown log: validate / DB write / cache-invalidation / total — analogous to the forward-bug diagnostic campaign: ship the probe to prod, read the runtime observation, then fix), OR stand up real perf observability first (see the `[RESEARCH]` observability item in inbox — distributed tracing would show the 10s breakdown directly). Diagnostics (#1171/#1173) and the dead `hydrateForwardedSnapshots` (#1170 — runtime-confirmed the re-fetch path never executed) removed post-verification in `c64ac9315`. Real-invariant tests kept: forward passthrough, content-threading, placeholder recovery, `NO_TEXT_CONTENT_PLACEHOLDER` value-lock._

_Cleared 2026-05-19:_

- _**Multi-personality ping race: wrong personality + duplicate response content** → resolved by **PR #1049** (shipped in v3.0.0-beta.123). Root cause: `ResponseOrderingService.processQueue` called a shared `deliverFn` for all drained results; the multi-tag closure ignored its arguments, so when two groups raced in the same channel one group's deliverFn re-delivered its own slots while the other group's deliverFn never fired. Fix: store `deliverFn` per `BufferedResult`. Originally observed 2026-05-17 in `#monotheism` channel — three @-pings to two personalities returned three Samael responses with the second text-identical to the first. Re-add this entry if the symptom resurfaces._

_Cleared 2026-05-10:_

- _**Cross-channel history bypassed `maxAge` and `contextEpoch` filters; cross-channel budget was a residual** → resolved by **PR #1011** (merged 2026-05-10, ships in next release). Three interlocking bugs: `getChannelHistory` ignored maxAge at the DB layer (stale rows leaked through, filling dbHistory to dbLimit); `fetchCrossChannelIfEnabled` computed cross-channel budget as `dbLimit - currentHistoryLength` (silently zero when current is full); `getCrossChannelHistory` had no time filter at all (would surface arbitrarily old context even after the other two fixed). Fix: shared `computeHistoryCutoff` helper used by both Prisma queries; cross-channel gets its own dbLimit budget (additive, not residual) — matches the user's mental model that cross-channel context is part of memory continuity, not "filler when current is sparse"._
