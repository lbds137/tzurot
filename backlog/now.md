**🐛 `[FIX]` z.ai coding-plan key not honored when resolving via the GLOBAL/admin default → routes via OpenRouter** — Surfaced 2026-06-29 (user report + screenshot). A user with a z.ai coding-plan key gets z.ai-direct routing when they SET their OWN default to a z.ai model (works — uses their z.ai quota), but when they rely on the **admin/global default** (also a z.ai model) the response footer reads `Model: z-ai/glm-4.7 · via OpenRouter` — auto-promotion to z.ai-direct does NOT fire, so their z.ai subscription quota is bypassed (served via OpenRouter instead). **Runtime-confirmed symptom** (screenshot footer). **Non-catastrophic** — gracefully falls back to OpenRouter (the user still gets a response), just on the wrong provider/quota; not a hard outage, so it needn't block the current epic. **Auto-promotion logic** (`ProviderRouter.ts`): promotes `provider==='openrouter'` + `isZaiCodingPlanModel(model)` + user-has-zai-coding-key → z.ai-direct. `glm-4.7` IS in the z.ai catalog (`constants/ai.ts:310`) and the user HAS a working key (their own-default path promotes), so **the bug is path-specific to global/admin-default resolution** — NOT the catalog or key validity. **Hypothesis (code-read, NOT yet runtime-confirmed which branch)**: the global-default path either (a) doesn't surface the requesting user's zai-coding key to `ProviderRouter`/`ApiKeyResolver` (promotion may be keyed to config OWNERSHIP — applied for the user's own config but not when the resolved config is the admin's global default), or (b) doesn't route the global-default path through `ProviderRouter` at all. **One diagnostic to confirm**: log `ProviderRouter` inputs (`provider`, `model`, `isZaiCodingPlanModel`, `hasZaiKey`) for a global-default resolution by a z.ai-key user → see which predicate is false. **Start**: `services/ai-worker/src/services/ProviderRouter.ts` (promotion predicate) + `AuthStep.ts` (caller) + `ApiKeyResolver.ts` (key resolution) + the LLM-config resolution cascade (user-default vs global/admin-default path). **UPDATE 2026-06-28 (new data point + bot-client trace, refines the framing)**: user reports the divergence tracks **trigger mode**, not (only) config source — a plain message in an **activated channel** promotes to z.ai (footer `glm-4.7 · via Z.AI Coding Plan 📍 auto`), but a **reply** to the persona does NOT (footer `z-ai/glm-4.7 · via OpenRouter`), same user + personality + model (runtime-confirmed, 2 screenshots). This may mean the earlier "global vs own-default" axis was a conflation; the controlled variable to isolate is **trigger mode with config source held constant**. **Bot-client trace (this session) RULES OUT a bot-client-side cause** (so the user's "bot-client DRY violation" hunch doesn't hold): reply (`ReplyResolutionService.resolvePersonality` → `HttpPersonalityLoader.loadPersonality`) and activation (`loadPersonalityStrict`) return the **same** `LoadedPersonality` (`loadPersonality` is just `loadPersonalityStrict` + error-reclassification, `HttpPersonalityLoader.ts:125`), and **both** trigger modes funnel through the SAME `PersonalityChatManager.submitChatJob` (`PersonalityChatManager.ts:206`), which uses `message.author` for BOTH the gateway client (`clientsForUser`) and `context.userId` (via `buildContext`). So `userId` + `personality` reaching the gateway are identical across modes — the bot-client submit layer is DRY-clean and does NOT explain the split. **Therefore the divergence is server-side or a runtime/data factor** (api-gateway `LlmConfigResolver` cascade / ai-worker `AuthStep` → `ProviderRouter` → `ApiKeyResolver`). Note the related latent issue [`cold/follow-ups.md` "ResolvedLlmConfig carries no provider" (#1239)]: promotion keys off the resolved **model name's `z-ai/` prefix**; both modes show `z-ai/glm-4.7` as the configured model, so the ONLY remaining lever is whether `tryResolveUserKey(userId)` resolves the key — yet `userId` is the same human in both. **This contradiction is the signal that a runtime factor is in play; do NOT ship a fix on code-reading alone.** **Sharpened diagnostic**: temporary `debug`-type log at `ProviderRouter.resolveRoute` entry capturing `{ configuredProvider, configuredModel, userId }` (the no-key / whitelist-miss branches already log at DEBUG — bump to INFO or enable debug on dev), reproduce a reply vs an activation for the same user+personality, and diff the three inputs. If `userId` differs → trace where the reply path overrides it server-side; if same but the key isn't found → `ApiKeyResolver` cache/scope; if `configuredProvider`/model differs → the cascade/seed-provider path. The reply also carries `rawReferencedMessages` (activation doesn't) — a candidate server-side variable worth checking in the same probe.
