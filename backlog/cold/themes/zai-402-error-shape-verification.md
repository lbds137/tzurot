### Theme: z.ai Catalog + 402 Error-Shape Verification

_Focus: empirically capture the real provider error shapes (z.ai catalog-drift 404s, z.ai 402 subscription-lapse, OpenRouter 402 credit variants), then narrow the deliberately-broad defensive code that shipped without samples._

Six sibling follow-ups from the z.ai-integration + credit-exhaustion-cache work (PRs #921/#924/#928/#945, late April 2026) share one blocker: we have no captured samples of the relevant provider error responses, so every defensive branch was intentionally shipped broad ("over-fallback is preferable to under-fallback while we have no error-shape data"). Each item keeps its own promote-when trigger, but they converge on the same unblocking event — a real probe or production incident that produces the missing error-shape data — and several name each other as the natural co-fix (item 5 explicitly says "alongside the z.ai 402-status verification work", i.e. item 6).

Consolidated from `backlog/cold/follow-ups.md` flat rows 2026-07-02 (granularity-ladder re-org — the rows were facets of one initiative).

#### 1. `isAccountCreditExhaustion` regex breadth

Fallback pattern `/insufficient.*credits/i` in `apiErrorParser.ts` is broader than the verbatim `/never purchased credits/i` arm; could match a hypothetical OpenRouter 402 with non-credit-exhaustion meaning that happens to mention "insufficient credits" (e.g., "insufficient platform credits for this tier"). Blast radius bounded today because `parseApiError` gates on `statusCode === 402 && resolved.category === ApiErrorCategory.QUOTA_EXCEEDED` — but if OpenRouter ever sends a 402 for a different reason with similar phrasing, it would get cached as account-level credit exhaustion and block subsequent calls for up to 24h. **Fix shape**: tighten the fallback to `/your account has insufficient credits/i` once we have a real 402-shape sample to anchor against, OR add an allowlist of explicit credit-exhaustion sub-patterns. **Promote when**: a 402 with non-credit-exhaustion meaning surfaces in production logs, OR opportunistically when next touching `apiErrorParser.ts`. Surfaced 2026-04-30 PR #945 round 3. Deferred 2026-05-01.

#### 2. 402 `max_tokens` auto-reduction (request-level credit-exhaustion fix)

When a 402 says "can only afford N tokens" (request-level, sub-classified as `QUOTA_EXCEEDED` not `CREDIT_EXHAUSTION`), the request COULD succeed with smaller `max_tokens`. Currently we just fail the user. A per-request retry with `max_tokens = M - safety_margin` would recover automatically. **Fix shape**: in `LLMInvoker`'s catch path, if 402 + `category === QUOTA_EXCEEDED` + technicalMessage carries an extractable token-budget number, retry once with reduced `max_tokens`. **Why deferred (strong reason)**: different mechanism from credit-exhaustion cache (per-request adjustment, not session-level memoization); no production evidence (10/10 production 402s observed 2026-04-29 were account-level "never purchased credits"); token-extraction from free-form provider messages is brittle (regex robustness, missing/malformed numbers, retry-loop bounding). **Promote when**: a user reports the request-level 402 pattern in production, OR the next time the credit-cache feature is touched and natural surface area exists. Surfaced 2026-04-30 PR #945 plan. Deferred 2026-05-01.

#### 3. Narrow auto-promotion fallback to catalog-drift errors specifically

`runWithAutoPromotionFallback` (`services/ai-worker/src/jobs/handlers/pipeline/steps/autoPromotionFallback.ts`) currently catches ANY error from the promoted z.ai request and retries via OpenRouter. Defensive but over-broad — a transient z.ai network blip, a temporary 5xx, or a rate limit will all silently route through OpenRouter without the user knowing. The actual catalog-drift signal is "z.ai returned 404 / model not found." **Fix shape**: once z.ai's API error format for "model not on plan" is empirically verified, narrow the catch to that specific error class. Other errors should propagate immediately so the user sees the real failure mode. ~5-10 LOC change. **Promote when**: we have a real z.ai error response captured (natural production usage, artificial probe with a known-bad model, or z.ai docs improvement). Until then the broad catch is the right starting point — over-fallback is preferable to under-fallback while we have no error-shape data. Surfaced 2026-04-28 PR #928 round 3. Deferred 2026-05-01.

#### 4. `ResolvedRoute.fallback` invariant via discriminated union (part b)

Part (a) — extracting a named `FallbackRoute` type alias — RESOLVED in PR #928; both `ResolvedRoute.fallback` and `ResolvedAuth.fallback` now reference the `FallbackRoute` interface. Part (b) remains: tighten the optional shape into a discriminated union so the `wasAutoPromoted === true ↔ fallback !== undefined` invariant becomes a static guarantee. **Fix shape (b)**: `BaseResolvedRoute & ({ wasAutoPromoted: true; fallback: FallbackRoute } | { wasAutoPromoted: false; fallback?: never })`. **Caveat**: needs to handle the graceful-degradation path where promotion succeeds but fallback computation fails — possibly a third union variant `{ wasAutoPromoted: true; fallback: FallbackRoute | undefined }`, which is just back to optional. May need rethinking once we have more data on how often the degradation fires. **Promote when**: feature has stabilized in production for a few weeks AND the graceful-degradation path is well-understood empirically. Surfaced 2026-04-28 PR #928 rounds 3+4. Deferred 2026-05-01.

#### 5. z.ai integration validation-model fallback (PR #921 follow-up)

PR #926 fixed the immediate day-1 bug (`glm-4.5-flash` was wrong — corrected to `glm-4.5-air`). Broader concern stands: if z.ai deprecates or renames `glm-4.5-air`, every valid key validates as `INVALID_KEY` (silent false-negative). **Fix shape options**: (a) try a list of fallback models in order (`glm-4.5-air` → `glm-4.7` → first-from-`/models`-endpoint); (b) hit a non-chat endpoint (e.g., `/account/balance` if z.ai exposes one — needs research); (c) accept the maintenance risk and add a periodic check to detect catalog changes early. **Promote when**: the integration scales enough that catalog change becomes a real risk, OR alongside the z.ai 402-status verification work. Surfaced 2026-04-27 PR #921. Deferred 2026-05-01.

#### 6. z.ai 402 HTTP status verification for expired/lapsed subscription

401/403/429 are accounted for in `mapValidationErrorToResponse`. If z.ai uses 402 for an expired coding-plan subscription, it falls through to `UNKNOWN` (api-gateway) or `InvalidApiKeyError` (ai-worker), neither of which conveys "subscription lapsed; renew" UX. **Research finding 2026-04-27**: z.ai's API docs at `docs.z.ai/api-reference/llm/chat-completion` only document HTTP 200 + a generic "default error" with structured `{ code, message }` body — they don't enumerate 401/402/403/404/429 explicitly. So a documentation-based answer is unobtainable; verification requires probing with a real expired/lapsed coding-plan key, which we don't have. **Fix shape**: when one becomes available (or when natural-occurrence of subscription expiration produces a real probe), capture the actual status code z.ai returns and surface "subscription lapsed" as a distinct error code in `mapValidationErrorToResponse`. Surfaced 2026-04-27 PR #921 round 3. Deferred 2026-05-01.
