### Theme: User-feedback solicitation + revive v2 release-notes delivery

_Focus: structured channel for soliciting feedback from non-direct-contact users — feedback today only arrives ad-hoc._

As broader UX issues come up (channel-activation behavior, voice handling, persona switching, etc.), a structured channel for hearing from users who aren't in direct contact with the operator would be valuable. Could double as the delivery path for the v2 release-notes feature (announce each release to interested users) that was lost in the v3 rewrite.

**Phases:**

1. **Opt-in/opt-out persistence** — likely a new column on the user row or a separate `user_preferences` table. Default-opt-in vs default-opt-out is the key design question.
2. **DM-blast job worker** — BullMQ scheduled job that iterates the opt-in cohort and sends a DM. Rate-limit-aware (Discord caps DMs from bots; bulk needs throttling). Idempotent retries on failure.
3. **`/admin broadcast` command surface** — bot-owner-only; accepts a message template + dry-run preview + opt-out compliance check before send.
4. **Release-notes integration** — wire the same delivery path to a CHANGELOG-driven announcement on release-PR merge (or manual trigger).

**Owner spec detail (ingested 2026-07-05 from notes-Discord cleanup — answers phase-1's key question)**: default is OPT-IN, opt-out via command, persisted. Granularity preference per user: notify on bugfix vs minor vs major version bumps (default: minor). Delivery mechanism sketch: bot stores last-announced version, compares against package.json at startup, DMs the delta's release notes on bump. Open sub-question from the note: fetching notes dynamically from the GitHub release tag (release notes already follow the machine-parseable Conventional Changelog format per `05-tooling.md` — that was built for exactly this).

**Abuse considerations (owner 2026-07-10 — design requirement, not optional)**: a `/feedback` surface is a spam/DoS vector ("potentially malicious people who wanna denial of service by spamming junk"). Minimums: per-user rate limit + cooldown (reuse `RateLimitCache` shape), hard length cap, per-user daily cap, and content lands in a queue/thread the owner reads on their terms (never a ping-per-submission). Consider dedupe on near-identical repeat submissions.

## Council-passed design (2026-07-14 — GLM 5.2 · Kimi K2.7-code · Qwen 3.7 Max; owner-decided taste calls)

**PROMOTED TO WORKING LINE 2026-07-14** (memory epic is observation-gated; this is the active build). Release-notes half ships first (owner priority, 2026-07-11).

**Resolved architecture** (unanimous council + repo-fact tiebreaks):

- **Trigger**: GitHub `release:published` webhook → api-gateway `/webhooks/github/release` (HMAC via the kept `GITHUB_WEBHOOK_SECRET`) — primary. Fallback = **scheduled reconcile** (BullMQ repeatable job comparing GitHub releases API against `release_announcements`), NOT boot-compare: Railway redeploys api-gateway at release-merge, BEFORE `release:publish` creates the GitHub release, so boot-compare fires when notes don't exist yet (project-specific fact that killed the v2 boot-fallback shape).
- **Storage**: columns on `users` (`notifyEnabled` default true, `notifyLevel` enum default minor) — not JSONB (blast query filters), not a prefs table (three fields). `release_announcements` (unique version = idempotency backbone; GitHub retries webhooks). `release_delivery_log` (release×user, status, error class) — subsumes per-user lastNotifiedVersion: retry-exactly-the-failures, failure classification, ops report, and no new-user-backfill trap.
- **Level policy (owner-decided)**: **changelog-derived** — Breaking Changes→major, Features→minor, fixes-only→patch, classified from the machine-parseable Conventional Changelog notes (built for exactly this). Beta bumps classify by CONTENT, not version delta — default-minor users get feature betas, skip fix-only ones.
- **DM worker**: bot-client BullMQ consumer (only service with the Discord client; JobFailureListener precedent). api-gateway resolves eligible recipients (bot-client stays Prisma-free) and enqueues batched jobs; limiter ~2 DMs/sec; `allowedMentions: parse: []`. Failure classification: 50007/blocked → permanent in delivery_log + auto-disable notifyEnabled after 2 consecutive permanents; transient → bounded retry. Delivery status reported back via gateway internal endpoint (established seam — no new ack queue).
- **Notes formatting**: own unit-tested module — truncate ~1800 chars, group by type, append GitHub release link.
- **Ops report**: after each blast, one silent post to the owner channel ("N sent, M permanent-failed (auto-opted-out), K transient").
- **Feedback intake**: `user_feedback` table (content, normalized content_hash, status enum). Gates enforced in api-gateway, cheap-first: length cap (2000) → cooldown (Redis TTL) → daily cap → normalized-hash near-dup dedupe vs user's recent rows. Ephemeral rejection names the specific limit. **Read surface (owner-decided): silent per-submission embed to a private owner channel** (pull, muted, zero pings); digest deferred until volume demands. Retention: status enum + eventual purge policy decided at build.
- **Dry-run/test path**: `/admin broadcast` dry-run lists eligible recipients without sending; webhook live-test is part of the release-pipeline PR's verification, not assumed.
