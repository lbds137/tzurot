### Phase 1 — Promote DEFERRABLE-FK harvesting to the shared PGLite schema generator (independent cleanup)

`generate-schema.ts` now harvests CHECK constraints + partial-UNIQUE indexes from migration SQL (the DDL Prisma's `migrate diff` can't represent), but NOT the `ALTER TABLE ... ALTER CONSTRAINT ... DEFERRABLE` statements. So deferrable FKs are absent from the shared `pglite-schema.sql`, and `DatabaseSyncService.component.test.ts` works around it bespokely (`loadDeferrableFkMigration()` + `SET CONSTRAINTS ALL DEFERRED`, asserting deferrability itself). **Fix shape**: add a third harvester (mirroring the CHECK + partial-unique extractors) for `ALTER CONSTRAINT ... DEFERRABLE` so the shared schema carries them, then drop `DatabaseSyncService.component.test.ts`'s bespoke migration-loading. This is a CLEANUP, not a coverage hole (the deferred-FK behavior IS tested today via the workaround). **Promote when**: next touching `generate-schema.ts` or the db-sync component test. Surfaced 2026-06-27 (PGLite-fidelity audit during vision-config epic).

### Phase 3 — Real-Postgres integration tests: trigger→NOTIFY cache-invalidation chain + concurrency + pool behaviors

A PGLite-fidelity audit (2026-06-27) found the behaviors PGLite structurally CAN'T model (vs. the schema-DDL gaps the harvester closes): (1) the `pg_notify` cache-invalidation **triggers + functions** (`notify_*_cache_invalidation()`) are absent from PGLite and untested at the component tier — the trigger→NOTIFY→`DatabaseNotificationListener`→cache-clear chain only works against real Postgres (PGLite is single-connection, no cross-connection LISTEN/NOTIFY); the listener's own side is unit-tested with mocks but the DB-trigger half isn't; (2) **concurrency/locking** — `SELECT FOR UPDATE`, advisory locks, serialization/deadlock, concurrent-upsert/`ON CONFLICT` races, MVCC-under-contention; (3) **`pg.Pool`** saturation / acquisition-timeout / the `DATABASE_POOL_STATS` gauge. Also low-value: prod's approximate IVFFlat **recall** behavior (PGLite seqscans → exact, so it can't surface a real match the index would miss). **Fix shape**: once the integration tier has real Postgres (Phase 2 above), add tests for the trigger/NOTIFY chain + the highest-risk concurrency path first. **Promote when**: the real-Postgres integration tier is stood up, OR a prod incident traces to one of these untested behaviors. Surfaced 2026-06-27 (PGLite-fidelity audit during vision-config epic).
