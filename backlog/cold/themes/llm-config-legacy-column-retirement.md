So this needs the **column-drop-prep** first (task #87). **Prep (release N) is a self-contained gateway type-refactor**: stop selecting (`LLM_CONFIG_LIST_SELECT`) + stop the `create` write (`LlmConfigService.ts:311`) + reconcile the type so `applyDefaultFlags`'s OUTPUT carries the derived flags rather than sourcing them from the select. **The sync util (`llmConfigSingletons.ts`) is NOT touched by the prep** — it selects the columns itself and still needs them for the partial-unique-index conflict resolution _while the columns exist_; its removal defers to the DROP release. **DROP (release N+1)**: `DROP COLUMN` + drop `llm_configs_default_unique`/`llm_configs_free_default_unique` partial indexes, THEN remove the now-obsolete sync util. **Verified non-issue**: the earlier worry that dev↔prod sync doesn't propagate the real defaults is false by design — `admin_settings` (the pointers) is env-specific, intentionally not synced (`syncTables.ts:18`). **Promote when**: opportunistically before a release, or when the DROP is wanted. ### Phase B — Drop the dormant `kind` column + per-kind indexes + `kind` plumbing

P3 leaves `LlmConfig.kind` MOSTLY dormant for capability/slot decisions (capability now decides slots via `supportsVision`; `AdminSettings` pointers replace the kind-slotted defaults) — BUT it is NOT fully dead: it is still WRITTEN on create (server-defaults to `text` when omitted) and READ as the **name-collision namespace** (`llm-config.ts` ~289–321: a vision rename collides only against vision names, never text). So the cleanup is not just "grep-confirm nothing references `kind`" — the name-collision scoping must first be **collapsed to a single namespace** (all configs share one name space) before the column can go. The cleanup migration then drops: the `kind` column, the per-kind partial-unique default indexes (`llm_configs_default_unique`, `llm_configs_free_default_unique`), and the now-unreferenced `toConfigKind`/`CONFIG_KINDS`/`ConfigKind` + `LlmConfigService`/`LlmConfigMapper` `kind` plumbing. **Fix shape**: collapse the name-collision check to ignore `kind`; a destructive migration (needs the maintenance-window mechanism — see the `MAINTENANCE_MODE` follow-up) + a grep-confirmed sweep that no code references `kind` before the drop. **Promote when**: Phase 3 (capability-driven) has shipped to prod and soaked clean. Surfaced 2026-06-29 (P3 plan — deferred cleanup; name-collision-reader nuance added during S4b-2 prep).
