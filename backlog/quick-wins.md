## ⚡️ Quick Wins

_Small tasks that can be done between major features. Good for momentum._

- **[CHORE] Migrate `/admin/db-sync` + `/admin/cleanup` to `X-User-Id` header** — both routes pass `{ownerId}` in POST body, which is why `extractOwnerId` retains a body-fallback path. All other admin routes use the `X-User-Id` header convention (post-PR #1081). Migrating these two would let the body fallback come out of `extractOwnerId` entirely. **Start**: `services/bot-client/src/commands/admin/db-sync.ts:137-140` + `commands/admin/cleanup.ts:31` — drop the `ownerId` body field, rely on `adminPostJson` setting the header. Then drop the body block in `services/api-gateway/src/services/AuthMiddleware.ts:extractOwnerId`. ~10 LOC. Surfaced by PR #1081 review.

- **[CHORE] Improve `cpd:check` drift error message to mention threshold mismatch** — when `cpd:check --threshold 0.7` runs against a baseline captured at the default `0.8`, the configHash mismatches and the tool fails with "Run `cpd:update-baseline` to refresh." A user who deliberately wants to check at a non-default threshold (without updating the baseline) gets misleading guidance — running `cpd:update-baseline` without `--threshold` re-captures at the default. **Start**: in `packages/tooling/src/commands/cpd.ts:runCheckCommand`, when emitting the drift error, also print "If you're running `cpd:check` with a non-default `--threshold`, confirm it matches the baseline's `threshold` field." Surfaced by PR #1084 round-7 review.

- **[CHORE] Add `--summary` mode to `guard:boundaries`** — `guard:boundaries` is registered as audit-class in `AUDIT_TOOL_REGISTRY`, but the command doesn't yet support `--summary`. Inconsistent with the other registered guards (`guard:proposal-links`, `guard:audit-tool-docs`), both of which emit JSONL summary lines. The Layer-5 aggregator (when built) will skip this tool until it does. **Start**: in `packages/tooling/src/commands/guard.ts:guard:boundaries`, add the `--summary` option matching the existing pattern. In `packages/tooling/src/dev/check-boundaries.ts`, gate the existing terminal output behind `!options.summary` and emit a JSONL line via `emitSummary` when `--summary` is passed (status: 'fail' if violations exist, 'ok' otherwise; findings count = violations.length). Surfaced by PR #1085 review (the doc inconsistency between 05-tooling.md and the registry table led to discovering this gap).

- **[CHORE] Add deterministic `orderBy` to `handleGetByResponse` `findFirst`** — `services/api-gateway/src/routes/admin/diagnostic.ts` `handleGetByResponse` calls `prisma.llmDiagnosticLog.findFirst({ where: { responseMessageIds: { has: messageId }, ... } })` without an explicit `orderBy`. Pre-existing behavior; not a regression. In practice response message IDs are unique per Discord message so the multi-row case is rare. **Start**: add `orderBy: { createdAt: 'desc' }` matching the pattern in `handleGetByMessage`. Surfaced by PR #1087 round-8 review.

- **[CHORE] Unify purge-confirmation-phrase case-sensitivity between bot-client and api-gateway** — `services/bot-client/src/commands/memory/purge.ts` (`handlePurgeModal`) compares the entered phrase against `expectedPhrase` with `!==` (case-sensitive). `services/api-gateway/src/routes/user/memoryBatch.ts` (`handleIssuePurgeToken`) compares with `.toUpperCase()` on both sides (case-insensitive). Since bot-client is the first gate, a user typing `"delete lilith memories"` (lowercase) fails at the client with a "phrase didn't match" message even though the API would have accepted it. The modal label implies uppercase-required, but the API is more permissive. **Start**: in `purge.ts` line ~360, change `if (enteredPhrase !== expectedPhrase)` to `if (enteredPhrase.toUpperCase() !== expectedPhrase.toUpperCase())` (the API's more user-friendly behavior). ~1 LOC. Surfaced by PR #1099 round-4 review.
- **[CHORE] Switch `batchDelete.ts` confirm/cancel button custom IDs to `::` delimiter** — `services/bot-client/src/commands/memory/batchDelete.ts` uses `memory_batch_delete_confirm` and `memory_batch_delete_cancel` (underscores), deviating from the `command::action::id` (`::`) convention in `.claude/rules/04-discord.md`. Fine today because these buttons are handled by an inline `awaitMessageComponent` collector, but if anyone later tries to route them through the standard `handleButton` export the custom ID won't parse. **Start**: rename both customIds to `memory-batch-delete::confirm` and `memory-batch-delete::cancel`, update the matching `customId === ...` checks in the same file. ~6 LOC. Surfaced by PR #1099 round-4 review.
- **[CUTOVER] Wallet rate-limiter must be re-applied when generated mounts replace legacy router** — `services/api-gateway/src/routes/wallet/index.ts:42` applies `createRedisWalletRateLimiter(redis)` at the router level (`router.use(...)`). The codegen mounts routes individually (`app.post(...)` etc.) and has no concept of router-scoped middleware, so when the cutover PR removes `app.use('/wallet', createWalletRouter(...))` and replaces it with the generated `mountUserRoutes(app, deps)`, the rate limiter will silently disappear. **Start**: in the cutover PR, either (a) add per-route rate-limiter middleware emission to the codegen (likely a new `RouteDef.rateLimiter?: 'wallet' | ...` field), or (b) keep a small `app.use('/api/user/wallet', createRedisWalletRateLimiter(redis))` line in `services/api-gateway/src/index.ts` before the mount call. **Promote when**: starting the route-prefix cutover PR (PR-2 in the original epic plan). Surfaced by PR #1097 round-8 review.
