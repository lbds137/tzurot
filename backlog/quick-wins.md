- 🧹 `[CHORE]` **Await async cleanup before `process.exit` in bot-client shutdown handler** — Surfaced 2026-04-26 by claude-bot review on PR #913. The shutdown sequence in `services/bot-client/src/index.ts` `handleShutdownSignal` calls `void client.destroy(); void disconnectPrisma(); void closeRedis();` then `process.exit(0)` synchronously. The voided promises don't resolve before exit — Discord WebSocket close handshake, Redis disconnect, and Prisma cleanup may not complete cleanly. Pre-existing pattern (PR #913 didn't introduce it, just refactored around it); flagged as the natural follow-up to the SIGTERM fix. The SIGTERM handler addressing the root cause makes this the next 20% of polish: with proper graceful shutdown, V1's gateway WebSocket sends a clean close frame before V2 finishes warming up, removing any residual orphan-session timing window. **Fix shape**: `await Promise.all([client.destroy(), disconnectPrisma(), closeRedis()])` before `process.exit(0)`. Add a max-wait timeout (e.g. 5s via `Promise.race` with a `setTimeout` reject) so a hung promise can't prevent shutdown — Railway will SIGKILL after a grace window anyway, better to exit cleanly first if we can. ~10 LOC. **Why deferred**: PR #913 was a fix-with-diagnostic shape, not a polish-while-you're-there shape. Land this after the diagnostic confirms PR #913 resolved the symptom, so the polish doesn't muddy the verification signal.
