## ⚡️ Quick Wins

_Small tasks that can be done between major features. Good for momentum._

- **[CHORE] Migrate `/admin/db-sync` + `/admin/cleanup` to `X-User-Id` header** — both routes pass `{ownerId}` in POST body, which is why `extractOwnerId` retains a body-fallback path. All other admin routes use the `X-User-Id` header convention (post-PR #1081). Migrating these two would let the body fallback come out of `extractOwnerId` entirely. **Start**: `services/bot-client/src/commands/admin/db-sync.ts:137-140` + `commands/admin/cleanup.ts:31` — drop the `ownerId` body field, rely on `adminPostJson` setting the header. Then drop the body block in `services/api-gateway/src/services/AuthMiddleware.ts:extractOwnerId`. ~10 LOC. Surfaced by PR #1081 review.

- **[CHORE] Detect orphan WHY.md files in `guard:audit-tool-docs`** — current check is unidirectional: every registered tool must have a WHY.md, but a WHY.md file with no registry entry would survive untouched (knip can't catch it — `whyPath` strings are data, not imports). **Start**: in `checkAuditToolDocsFromRegistry`, walk `packages/tooling/src/**/*.WHY.md` and assert each path matches at least one registry entry. Add a third finding category (`orphanWhyFiles`) reported as hard-fail. Surfaced by PR #1083 review.

- **[CHORE] Rename `singleSegmentSlugs` → `singleSegmentProposals`** — the field in `OrphanCheckResult` contains relative paths (e.g., `docs/proposals/backlog/memory.md`), not bare slugs. Current name misleads readers of the interface. Trivial rename — touches the type declaration, the assignment site in `findProposalOrphans`, the CLI consumer, and the tests. **Start**: `packages/tooling/src/audits/check-proposal-orphans.ts:24-29` (the `OrphanCheckResult.singleSegmentSlugs` field) plus all references. Surfaced by PR #1083 post-merge review.

- **[CHORE] Add a one-line comment on the lazy quantifier in the frontmatter strip regex** — `^---\r?\n[\s\S]*?\r?\n---[^\n]*(?:\r?\n|$)` uses `[\s\S]*?` (lazy match). If a WHY.md frontmatter scalar contained an embedded `---` line, the regex would treat it as the closing fence. The behavior is correct for the project's WHY.md format, but a future reader might wonder why eager wasn't chosen. **Start**: one inline comment next to the regex in `check-audit-tool-docs.ts`. Surfaced by PR #1083 post-merge review.

- **[CHORE] Reconsider case-sensitivity of orphan-check regex for SCREAMING_SNAKE_CASE proposals** — the word-boundary regex in `findProposalOrphans` is case-sensitive by intent (multi-segment kebab-case is the documented convention). The codebase already has 5 legacy SCREAMING_SNAKE_CASE proposals; if a future contributor links one in prose as `git-hook-improvements` instead of `GIT_HOOK_IMPROVEMENTS`, it would silently flag as an orphan. **Start**: either add the `'i'` flag back to the regex (safe — multi-segment names are precise enough that case-insensitive matches don't increase false-negative risk), OR surface the case requirement explicitly in the human-readable error output. Latent — current real-repo state has no case-mismatched mentions. Surfaced by PR #1083 post-autosquash review.
