- 🧹 `[CHORE]` **Make `guard:duplicate-exports` and `knip` checks blocking in CI** — Surfaced 2026-04-27 during PR #918 work. Both checks currently have `continue-on-error: true` in `.github/workflows/ci.yml` so they print warnings but don't fail the lint job. Result: real findings get treated as informational and easily missed (e.g., PR #918's first push had `StartupDMPrewarmerDeps` flagged by knip and would have been merged uncaught if the contract-audit hadn't independently caught the schema gap). **Why deferred**: enforcement requires building an allowlist for legitimate duplicates first. As of 2026-04-27 the duplicate-exports check shows 10 groups — most are intentional (mocks under `__mocks__/` duplicating production exports per Vitest pattern, re-exports through `index.ts` barrels). Triaging them is a focused 30-60 min task. **Fix shape**: (a) audit `pnpm ops guard:duplicate-exports` output, classify each group as legitimate/incidental, (b) extend the script's allowlist (`packages/tooling/src/dev/check-duplicate-exports.ts` `ALLOWLIST` constant) to cover the legitimate ones, (c) audit `pnpm knip` output the same way, configure `knip.json` ignores for intentional internal exports, (d) flip `continue-on-error: true` → `false` (or remove the line) in both CI steps. Once enforced, reviewers + CI both gain a real signal on dead code and unintended duplicates.

- 🧹 `[CHORE]` **Delete `DM_RAW_GATEWAY_DIAGNOSTIC` env var entry from Railway dashboard** — Set to `false` on 2026-04-27 after empirical confirmation Layer 1 + Layer 2 fix the post-deploy DM-silence bug (PR #918). - 🧹 `[CHORE]` **Await async cleanup before `process.exit` in bot-client shutdown handler** — Surfaced 2026-04-26 by claude-bot review on PR #913. The shutdown sequence in `services/bot-client/src/index.ts` `handleShutdownSignal` calls `void client.destroy(); void disconnectPrisma(); void closeRedis();` then `process.exit(0)` synchronously. The voided promises don't resolve before exit — Discord WebSocket close handshake, Redis disconnect, and Prisma cleanup may not complete cleanly. Pre-existing pattern (PR #913 didn't introduce it, just refactored around it); flagged as the natural follow-up to the SIGTERM fix. The SIGTERM handler addressing the root cause makes this the next 20% of polish: with proper graceful shutdown, V1's gateway WebSocket sends a clean close frame before V2 finishes warming up, removing any residual orphan-session timing window. **Fix shape**: `await Promise.all([client.destroy(), disconnectPrisma(), closeRedis()])` before `process.exit(0)`. Add a max-wait timeout (e.g. 5s via `Promise.race` with a `setTimeout` reject) so a hung promise can't prevent shutdown — Railway will SIGKILL after a grace window anyway, better to exit cleanly first if we can. ~10 LOC. **Why deferred**: PR #913 was a fix-with-diagnostic shape, not a polish-while-you're-there shape. Land this after the diagnostic confirms PR #913 resolved the symptom, so the polish doesn't muddy the verification signal.

- 🧹 `[CHORE]` **Replace local `CreatePersonaResponse` interface in bot-client with imported common-types type** — Surfaced 2026-04-26 by claude-bot review on PR #912. `services/bot-client/src/commands/persona/create.ts:25-35` declares a local `CreatePersonaResponse` interface that duplicates the Zod-validated schema in `packages/common-types/src/schemas/api/persona.ts`. The duplication means the local interface and the canonical schema can drift without a compile-time check — exactly what just happened with the `setAsDefault` field (removed from schema in PR #912 but the local interface had to be hand-edited too). **Fix shape**: `import type { CreatePersonaResponse } from '@tzurot/common-types';` (the schema-inferred type), then delete the local interface. ~5 min. **Why deferred from PR #912**: PR #912 was already approved with LGTM and the type-import refactor is a separate concern (drift-prevention vs. dead-pipe removal); folding it in would expand scope post-approval.
