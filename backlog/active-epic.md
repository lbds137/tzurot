#### The design fork (council pass required before code)

1. **Endpoint-mirroring** â€” one internal route per operation (~12â€“14 new routes). Mechanical, smallest per-PR risk, but converts 4â€“8 in-process DB queries per message into 4â€“8 HTTP round-trips per message. Latency cost on the 3-second-sensitive path is the core objection.
2. **Context-assembly relocation** â€” move context building server-side: bot-client submits thin references (channelId, userId, discordMessageIds); ai-worker (which legitimately owns Prisma) hydrates context inside the job. Eliminates most Cluster A endpoints entirely; biggest restructure (MessageContextBuilder largely dissolves into ai-worker's ContextStep, which already does context work â€” possible duplication-collapse win). Persistence (Cluster B) could similarly move into ai-worker post-generation + a small sync endpoint for SyncExecutor.
3. **Hybrid (likely council favorite, prejudging slightly)** â€” ONE batched internal `assembleMessageContext` endpoint for Cluster A (1 round-trip per message, server does the 4â€“8 queries), POST endpoints for Cluster B persistence/sync, `loadPersonality` as a cached internal route (bot-client's TTL caches stay, only misses pay the HTTP hop). Smaller restructure than (2), bounded latency unlike (1).

#### Council verdict â€” 2026-06-04 (GLM-5.1 + Kimi-K2.6 + Qwen-3.7-max, UNANIMOUS on the fork)

**(a) Fork 2 â€” context-assembly relocation â€” unanimous.** Shared reasoning: (1) the latency objection to endpoint-mirroring was misframed â€” context reads happen pre-job-submission but generation is async anyway, so moving reads INTO the job removes them from pre-submission latency entirely; (2) it collapses the existing MessageContextBuilder/ContextStep duplication (subtractive refactor â€” delete, don't rewrite); (3) Fork 3's batched endpoint is a god-route that "preserves the architectural confusion" (Qwen) and teaches api-gateway AI-domain context logic it shouldn't know (GLM).

**(b)** Context assembly lives entirely inside ai-worker's `ContextStep` â€” no separate hydration step ("splitting them creates a DTO that's essentially the assembled context, which means assembling twice under different names" â€” Qwen). bot-client submits a thin `RawDiscordEnvelope` (channelId, guildId, userId, content, discordMessageIds, attachments, author metadata). **(c)** `getOrCreateUser` upsert â†’ inside ai-worker hydration (idempotent, on the async path) â€” unanimous.

**(d)** Split by event ownership (GLM + Qwen majority; Kimi wanted everything in ai-worker but api-gateway-as-data-authority matches our architecture rules): **generation persistence â†’ ai-worker** (it has the full turn: prompt, response, tokens, model â€” writes atomically at job end); **Discord-event writes â†’ bot-client POSTs to api-gateway** (edit sync, delete sync, delivery confirmation â€” bot-client is the only service that sees these events, and Qwen caught the dependency inversion: ai-worker can't know the Discord message ID, which only exists after delivery). **~3 narrow internal endpoints total**, not 12-14.
