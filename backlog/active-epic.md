### Phase 1 ‚Äî Vision as a first-class config (‚úÖ SHIPPED ‚Äî PR #1364, beta.140)

Vision model decoupled from `LlmConfig` via a `kind: 'text' | 'vision'` discriminator: vision presets are `kind='vision'` rows whose `model` IS the vision model; `visionModel` removed from `LlmConfig`; `personality.visionModel` kept as the in-memory carrier, filled gateway-side by `VisionConfigResolver` (mirrors `TtsConfigResolver`). Schema migration (kind + per-kind partial-unique default indexes + vision FK/join columns), `VisionConfigBootstrap` seed, per-kind default/list/name-check scoping in `LlmConfigService`, surface amputation. Vision is **seed/DB-only ‚Äî no editing UI yet** (that's Phase 2).

### Phase 3 ‚Äî Capability-driven config model (IN PROGRESS)

**Pivot (2026-06-29):** testing the Phase-2 browse surface exposed that the `kind: 'text'|'vision'` discriminator is the wrong abstraction ‚Äî it caused the `/preset global free-default kind:Vision` "Config not found" bug, a misleading üëÅ badge (showed config kind, not model capability ‚Äî Gemma 4 supports vision but showed nothing), and duplicate configs for one model. Re-architected so a config is `{name, model, params}` with NO user-facing kind; the model's **capability** (OpenRouter-authoritative ‚Üí z.ai catalog ‚Üí reject-on-unknown, via `ModelCapabilityService`) decides slot-eligibility. Two slots (chat, vision) resolve through one uniform cascade; the vision slot is capability-gated; unknown capability = unusable model ‚Üí rejected. `kind` column left DORMANT (Phase B drops it).
