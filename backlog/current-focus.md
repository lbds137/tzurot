2. **adminFetch + admin-routes architectural refactor** — QUEUED, follows item 1 immediately. Council-recommended fix for the recurring "bot-client forgot to send X-User-Id" footgun:
   - Split client-side `adminFetch(path, { userId? })` → `serviceFetch(path)` (service-only, no user header) + `userFetch(path, userId, options)` (userId REQUIRED positional, both headers). TS-enforced intent at every call site.
   - Reorganize server-side route prefixes: `/api/internal/*` (requireServiceAuth mounted globally), `/api/admin/*` (requireOwnerAuth), `/api/user/*` (requireUserAuth). Route prefix carries the auth contract; no per-route ad-hoc middleware.
   - This is a system-wide migration touching every existing admin route + every bot-client call site. Needs its own council pass on the migration plan. Filing as Active Epic candidate once item 1 ships — likely 2-3 phased PRs (introduce new helpers + prefix mounts → migrate routes → delete legacy adminFetch + old prefix mounts).
   - **Surfaced during PR #1087 round-2 review**: PATCH `/admin/diagnostic/:requestId/response-ids` swapped from `requireOwnerAuth` to `requireServiceAuth`, which removed the human-owner constraint and replaced it with "any holder of INTERNAL_SERVICE_SECRET can update any diagnostic log row." Realistic blast radius is bounded (cannot create phantom rows due to P2025 on missing, cannot read other users' data), and bot-client is the only legitimate caller. Trust assumption documented inline in `services/api-gateway/src/routes/admin/diagnostic.ts`. The route-prefix refactor will address this structurally by giving service-only routes a distinct prefix + middleware path — the implicit "this is internal-only" invariant becomes an explicit URL-level contract instead of a per-route middleware choice.
   - **Also surfaced during PR #1087 round-2 review**: the GET diagnostic routes' service-secret enforcement is provided by the global `requireServiceAuth` mounted in `index.ts`, not by per-route middleware. If routes are ever extracted to a sub-app or alternate mount path that bypasses the global guard, callers could query diagnostic logs with only `X-User-Id`. Latent architectural risk, not a current bug. Same refactor scope addresses this via explicit `/api/internal/*` vs `/api/user/*` prefix semantics.
   - **PR #1087 round-6 — forensic audit trail for non-owner diagnostic-log lookups**: The pre-PR client-side filter emitted `[Inspect] Access denied: log belongs to another user` whenever a non-owner tried to look up a log that existed but didn't belong to them — the only signal for detecting enumeration attacks (user probing incrementing requestIds). The new server-side 404-not-403 design correctly hides existence but loses this forensic capability — `handleGetByRequestId`'s extended-WHERE `findUnique` returns null for both "doesn't exist" and "exists but not yours," so we can't distinguish at the handler. Reviewer's suggested mitigation (debug log on every non-owner 404) is a noisy false-positive tradeoff vs. zero signal. The route-prefix refactor is a natural place to revisit this with a proper threat-model design pass — e.g., a secondary count-only query for non-owner 404s, sampled to avoid log noise, or a separate structured audit log table. **Promote when**: enumeration probing is observed (or a feature requires forensic visibility into denied diagnostic accesses).
   - **PR #1087 round-6 — redundant per-route `requireServiceAuth` on PATCH `/response-ids`**: The global `app.use(requireServiceAuth())` in `services/api-gateway/src/index.ts` already validates `X-Service-Auth` for all `/admin/*` routes before reaching the per-route middleware. The PATCH route's per-route `requireServiceAuth()` is a double-application — harmless today but a maintenance trap if routes are ever extracted to a sub-app where the per-route guard might give a false sense of being "the" guard. Same refactor scope: route-prefix reorg naturally cleans up the duplication because service-only routes will live under `/api/internal/*` with the auth middleware mounted at the prefix, not per-route.
